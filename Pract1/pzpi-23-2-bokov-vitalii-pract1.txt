3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Короткі відомості про патерни проєктування
Патерни проєктування – це перевірені часом рішення типових проблем,
що виникають при проєктуванні програмного забезпечення. Вони не є готовими
фрагментами коду, які можна просто скопіювати, а представляють собою
концептуальні схеми взаємодії об’єктів та класів.
За класифікацією «Банди чотирьох» («Gang of Four», «GoF») патерни
поділяються на три групи:
а) твірні (creational) – створення об’єктів;
б) структурні (structural) – зв’язки між об’єктами;
в) поведінкові (behavioral) – взаємодія об’єктів та обов’язки.
Використання патернів дозволяє зменшити складність системи за рахунок
перевірених архітектурних рішень, підвищити її гнучкість, масштабованість та
повторне використання коду.
3.2 Опис обраного патерна
Обраний патерн – «Шаблонний метод» (Template Method) – поведінковий
патерн проєктування, який визначає скелет (каркас) алгоритму в базовому
класі, делегуючи реалізацію окремих його кроків підкласам. Підкласи можуть
перевизначати певні кроки алгоритму, не змінюючи його загальної структури
та послідовності виконання.
Послідовність алгоритму визначається один раз у батьківському класі та
зберігається незмінною, тоді як підкласи вільно налаштовують лише ті кроки,
що відрізняються між реалізаціями.
Патерн вирішує проблему дублювання коду, що виникає, коли кілька
класів реалізують той самий процес із незначними відмінностями:
а) дублювання коду – кілька класів, що реалізують один і той самий
процес, призводять до повторюваного та складного в підтримці коду;
б) крихкі зміни – зміна загальної логіки вимагає окремого редагування
кожного класу;
в) схильність до помилок – однакові кроки, що підтримуються в різних
класах, підвищують ризик неузгодженостей і багів.
Використання патерна Шаблонний метод є доцільним у таких випадках:
а) коли алгоритм має стабільну послідовність кроків, але окремі його
кроки відрізняються між реалізаціями (фіксована структура, змінні
деталі);
б) коли схожі класи мають більшість спільної логіки та відрізняються
лише кількома ключовими операціями, тобто коли потрібно усунути
дублювання;
в) коли потрібно дозволити розширення лише певних етапів процесу, а
не всього алгоритму (контрольована розширюваність).
Переваги:
а) централізований контроль – зміни в базовому класі поширюються на
всю систему: одне редагування оновлює поведінку всюди;
б) повторне використання коду – спільну логіку винесено до одного
суперкласу, що усуває повторення між підкласами;
в) сувора структура – точки розширення обмежені лише тими методами,
які явно дозволено, без несанкціонованих перевизначень;
г) гнучкість завдяки хукам – необов’язкові методи дозволяють підкласам
вбудовуватися в алгоритм у певних точках, не перевизначаючи всіх
кроків.
Недоліки:
а) клієнти можуть бути обмежені наданим скелетом алгоритму, оскільки
загальна структура задається базовим класом;
б) складніше дотримуватися принципу підстановки Лісков, якщо підклас
випадково придушує крок, передбачений базовою реалізацією;
в) шаблонні методи стає важче супроводжувати, що більше кроків вони
містять.
3.3 Структура патерна
Патерн Шаблонний метод ґрунтується на механізмах успадкування та
поліморфізму. Базовий (абстрактний) клас містить шаблонний метод, який
задає послідовність кроків алгоритму, а також оголошує набір методів-кроків,
частина з яких є абстрактними та реалізується у підкласах.
Структура патерна включає такі основні елементи:
а) Template Method (шаблонний метод) – визначає послідовність кроків
алгоритму; зазвичай оголошується як final, щоб підкласи не могли
змінити порядок його виконання;
б) Primitive Methods (примітивні операції) – абстрактні кроки, які
підкласи зобов’язані реалізувати, надаючи власну специфічну
поведінку;
в) Concrete Methods (конкретні операції) – спільні дії, однакові для всіх
підкласів; реалізуються один раз у базовому класі;
г) Hooks (хуки) – необов’язкові методи з порожньою або типовою
реалізацією, які дозволяють підкласам «вбудовуватися» в алгоритм у
певних точках, не перевизначаючи всіх кроків.
Базовий клас містить фіксований шаблон – структура алгоритму ніколи
не змінюється, а підкласи надають лише реалізації абстрактних методів для
змінних кроків, зберігаючи загальний потік незмінним. Саме тому шаблонний
метод зазвичай оголошують зі специфікатором final, щоб запобігти зміні
послідовності алгоритму підкласами.
Рисунок 3.1 – Структура патерна Шаблонний метод
На рис. 3.1 зображено структуру патерна Шаблонний метод. Абстрактний
клас AbstractClass містить шаблонний метод templateMethod(), який у
фіксованому порядку викликає примітивні операції. Конкретні підкласи
ConcreteClass успадковують шаблонний метод без змін та перевизначають лише
абстрактні примітивні операції, реалізуючи специфічні кроки алгоритму.
3.4 Реалізація патерна
3.4.1 Опис реалізації
Для впровадження патерна Шаблонний метод спочатку аналізують
цільовий алгоритм і розбивають його на послідовність окремих кроків. Кроки,
спільні для всіх варіантів, реалізують безпосередньо в базовому класі, а ті, що
відрізняються між реалізаціями, оголошують абстрактними.
Далі в базовому класі створюють шаблонний метод, який у фіксованому
порядку викликає всі кроки алгоритму. Цей метод доцільно оголосити як final,
щоб підкласи не могли порушити послідовність виконання. За потреби додають
хуки – методи з типовою реалізацією, які підкласи можуть перевизначати за
бажанням. Нарешті, конкретні підкласи реалізують абстрактні кроки, не
змінюючи загальної структури алгоритму.
3.4.2 Приклад на мові програмування TypeScript
Розглянемо систему керування кешем. Обидві стратегії – FIFO (першим
прийшов – першим вийшов) та LRU (найменш нещодавно використаний) –
мають однаковий алгоритмічний каркас: отримати значення за ключем,
перевірити його наявність та повернути результат. Відрізняються вони лише
одним критичним кроком – способом витіснення елемента при переповненні
кешу: FIFO видаляє найстаріший запис, а LRU – той, що використовувався
найменш нещодавно.
Без патерна Шаблонний метод логіку отримання, перевірки та
повернення значень довелося б дублювати в кожному класі кешу, що
ускладнює підтримку та підвищує ризик неузгодженостей між реалізаціями.
Базовий клас Cache [додаток В.1] містить шаблонний метод get(), який
задає незмінну послідовність кроків, та абстрактний метод evict() для
витіснення елемента. Підкласи FIFOCache та LRUCache [додаток В.2]
перевизначають крок витіснення evict() (а LRU — також хук відстеження
доступу touch()), реалізуючи власну стратегію, тоді як решта алгоритму
залишається спільною. Завдяки цьому дублювання коду усувається, а
додавання нової стратегії не потребує зміни наявного коду.

Приклади програмного коду
В.1 Базовий клас із шаблонним методом (Cache)
01 abstract class Cache {
02 protected store = new Map<string, number>();
03 constructor(protected capacity: number) {}
04
05 // Template Method — незмінний скелет алгоритму
06 get(key: string): number | undefined {
07 const value = this.store.get(key);
08 if (value === undefined) return undefined;
09 this.touch(key); // Hook
10 return value;
11 }
12
13 put(key: string, value: number): void {
14 if (this.store.size >= this.capacity) {
15 this.store.delete(this.evict()); // Primitive
16 }
17 this.store.set(key, value);
18 this.touch(key);
19 }
20
21 // Primitive Method (реалізують підкласи)
22 protected abstract evict(): string;
23
24 // Hook — типова реалізація порожня
25 protected touch(key: string): void {}
26 }
В.2 Підкласи зі стратегіями витіснення (FIFOCache, LRUCache)
01 // FIFO — витісняє найстаріший запис
02 class FIFOCache extends Cache {
03 private order: string[] = [];
04
05 protected evict(): string {
06 return this.order.shift()!;
07 }
08
09 protected touch(key: string): void {
10 if (!this.order.includes(key)) this.order.push(key);
11 }
12 }
13
14 // LRU — витісняє найменш нещодавно вживаний
15 class LRUCache extends Cache {
16 private usage: string[] = [];
17
18 protected evict(): string {
19 return this.usage.shift()!;
20 }
21
22 protected touch(key: string): void {
23 this.usage = this.usage.filter((k) => k !== key);
24 this.usage.push(key);
25 }
26 }
27
28 // Використання
29 const fifo: Cache = new FIFOCache(2);
30 fifo.put("a", 1);
31 fifo.put("b", 2);
32 fifo.get("a");
